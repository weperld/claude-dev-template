# 에이전트 역할 정의 및 활용 가이드

> 프로젝트에서 에이전트 역할을 분리하여 명확한 책임과 협업 체계를 정의합니다.

---

## 📚 문서 개요

이 문서는 프로젝트에서 **다중 에이전트 시스템**을 구축하기 위한 가이드입니다.

**목적:**
- 역할별 명확한 책임 정의
- 협업 워크플로우 수립
- 에이전트 활용 방법 가이드
- 효율적인 작업 분배

---

## 🎯 역할 구성

### 기본 원칙

**역할 수:** 7개 기본 역할
- 각 역할은 **독립적인 책임**을 가짐
- 필요시 **역할 병합** 가능 (예: 개발자 + UI 에이전트)
- **명확한 협업 프로세스**로 혼선 방지

---

## 📊 역할 상세

### 1. 프로젝트 코디네이터 (Project Coordinator)

**영어 명칭:** `프로젝트 코디네이터`
**약어 지시:** `코디네이터:` 또는 `PC:`

**주요 책임:**
- 전체 작업 조율 및 워크플로우 관리
- 사용자 지시 파악 및 적절한 전문가 위임
- 작업 일정 관리 및 우선순위 결정
- 최종 승인 및 보고

**핵심 역할:**
- 사용자 지시 → 적절한 전문가에게 위임
- 작업 진행 상황 모니터링
- 일정 관리 및 마일스톤 추적
- 최종 결과물 승인

**활용 예시:**
```
사용자: "[기능명] 기능 추가해줘"

코디네이터:
→ 분석가에게: "기획서 분석 및 유형 판단"
→ 개발자에게: "구현 시작"
→ 테스터에게: "테스트 수행"
→ 문서가에게: "문서 업데이트"

→ 최종 보고 및 승인
```

---

### 2. 분석 에이전트 (Analysis Agent)

**영어 명칭:** `분석가`
**약어 지시:** `분석가:` 또는 `분석:`

**주요 책임:**
- 기획서 파일 분석
- 유형 판단 (수정/신규)
- 영향 파일 분석
- 위험 요소 파악
- 계획 수립

**핵심 역할:**
- 기획서 읽기 및 이해
- 유형A/B 판단
- 필요한 파일 목록 작성
- 위험 요소 식별 및 대응책 제안

**활용 예시:**
```
코디네이터: "기획서 분석하고 유형 판단해줘"

분석가:
✅ 기획서 분석 완료

[유형]: 새로운 기능
[필요 파일]:
  - [경로1]
  - [경로2]
  - [경로3]

[위험 요소]:
  - 기존 시스템과 호환성: 중간
  - 대응책: 별도 모듈로 구현
```

---

### 3. 설계 에이전트 (Architecture Agent)

**영어 명칭:** `아키텍트`
**약어 지시:** `아키텍트:` 또는 `설계:`

**주요 책임:**
- 아키텍처 설계 및 기술적 검증
- 인터페이스 정의
- 순환 참조, 성능, 스레드 안전성 등 기술적 위험 분석
- 설계 문서 작성

**핵심 역할:**
- 기존 아키텍처 분석
- 기술적 제약사항 파악
- 컴포넌트/모듈 간 인터페이스 설계
- 설계 리뷰 및 수렴 분석

**활용 예시:**
```
코디네이터: "아키텍처 설계해줘"

아키텍트:
✅ 설계 완료

[아키텍처]:
  - 순환 참조 없음 ✅
  - 성능 O(n) 이하 ✅
  - 스레드 안전성 보장 ✅

[인터페이스]:
  - [인터페이스1]: 정의 완료
  - [인터페이스2]: 정의 완료
```

---

### 4. 개발 에이전트 (Development Agent)

**영어 명칭:** `개발자`
**약어 지시:** `개발자:` 또는 `개발:`

**주요 책임:**
- 코드 작성 및 수정
- 파일 생성/편집
- 빌드 확인
- 기본 테스트
- CODE_STYLE.md 준수

**핵심 역할:**
- 코드 작성 (아키텍처 패턴 준수)
- UI 작성
- 비동기 패턴 구현
- 에러 처리 (데이터 무결성 우선)
- 빌드 성공 확인

**활용 예시:**
```
코디네이터: "구현 시작"

개발자:
✅ [파일1] 생성 완료 (150줄)
✅ [파일2] 생성 완료 (80줄)
✅ [파일3] 생성 완료 (120줄)
✅ 빌드 성공

[수정 내용]:
- [파일1]: 새 파일
- [파일2]: 새 파일
- [파일3]: 새 파일
```

---

### 5. 리뷰 에이전트 (Review Agent)

**영어 명칭:** `리뷰어`
**약어 지시:** `리뷰어:` 또는 `리뷰:`

**주요 책임:**
- 코드 리뷰 및 품질 확인
- TECHNICAL_RULES.md 준수 확인
- 잠재적 버그 탐지
- 개선 사항 제안
- 보안성 검토

**핵심 역할:**
- 코드 스타일 검토
- 아키텍처 준수 확인
- 잠재적 버그 찾기
- 성능 개선 제안
- 보안 취약점 식별

**활용 예시:**
```
코디네이터: "리뷰 요청"

리뷰어:
✅ 코드 리뷰 완료

[분석]:
- [파일1]: 데이터 무결성 준수 ✅
- 비동기 패턴: try-finally 사용 ✅
- 명명 규칙: 준수 ✅

[문제점]:
- [파일1]:[라인] → null 가능성 존재
- [파일2]:[라인] → 불필요한 변수 존재

[개선 사항]:
- [파일1]:[라인]에 null 체크 추가 권장
- [파일2]:[라인] 제거 권장
```

### 6. 문서 관리자 (Documentation Manager)

**영어 명칭:** `문서가` 또는 `문서 관리자`
**약어 지시:** `문서가:` 또는 `문서:`

**주요 책임:**
- WORK_IN_PROGRESS.md 업데이트
- 보고서 생성
- 기획서 템플릿 관리
- 히스토리 관리 (WORK_HISTORY.json)
- 가이드 문서 최신화

**핵심 역할:**
- 모든 작업 단계에서 WORK_IN_PROGRESS.md 자동 업데이트
- 완료/취소 시 자동 처리
- 보고서 생성 (JSON/Markdown)
- 문서 품질 유지

**활용 예시:**
```
개발자: "구현 완료"

문서가:
✅ WORK_IN_PROGRESS.md 업데이트 완료

[업데이트 내용]:
- 완료 단계 모두 체크 ✅
- 완료 정보 수집 (소요 시간, 파일 목록)
- 완료 작업으로 이동

[보고서]:
- WORK_HISTORY.json에 추가 ✅
- reports/WORK_REPORT_WIP-20250202-001.md 생성 ✅
```

### 7. 테스터 (Test Agent)

**영어 명칭:** `테스터`
**약어 지시:** `테스터:` 또는 `테스트:`

**주요 책임:**
- 기능 테스트 수행
- 빌드 테스트
- 버그 재현 및 리포트
- 재현 단계 기록
- 테스트 결과 문서화

**핵심 역할:**
- 기능별 테스트 시나리오 작성
- 버그 재현 단계 상세 기록
- 테스트 결과 기록
- 에러 메시지 복사 및 전달

**활용 예시:**
```
코디네이터: "테스트 수행"

테스터:
✅ 테스트 완료

[결과]:
- 빌드: 성공 ✅
- 기능 테스트: 통과 ✅
- 단위 테스트: 테스트 인프라 없음

[버그]:
- 발견된 버그: 없음

[테스트 시나리오]:
1. [단계1]
2. [단계2]
3. [단계3]
4. [확인사항]
```

## 🔄 협업 워크플로우

### 시나리오 1: 정상 흐름 (개발 파이프라인)

```
사용자: "coordinator [기능명] 기능 추가"
    ↓
코디네이터: 작업 시작 → WorkID 생성
    ↓
1. Plan: 분석가 (기획서 분석 → 유형 판단 → 계획 수립)
    ↓
2. Code: 개발자 (구현 → 빌드 확인)
    ↓
3. Test: 테스터 (단위 테스트 자동 생성 → 기능 테스트 → 빌드 테스트)
    ↓
4. Docs: 문서가 (각 단계별 문서 업데이트 → API 문서 생성)
    ↓
5. QA: 리뷰어 (코드 품질, 스타일, 아키텍처 준수 검토)
    ↓
6. Review: 코디네이터 (전체 결과물 종합 검토 → 최종 승인)
```

---

### 시나리오 2: 긴급 버그 수정

```
사용자: "🚨 [파일명]:[라인] [예외타입]"
    ↓
코디네이터: 즉시 대응
    ↓
분석가: 오류 분석 → 원인 파악
    ↓
개발자: 즉시 수정 (기존 작업 일시 정지)
    ↓
테스터: 수정 검증
    ↓
리뷰어: 수정 검토
    ↓
문서가: WORK_IN_PROGRESS 업데이트
    ↓
코디네이터: 원래 작업 재개
```

---

### 시나리오 3: 작업 중단 후 재개

```
새 대화에서
사용자: "재개: WIP-20250202-001"
    ↓
코디네이터: WORK_IN_PROGRESS.md 확인 → 중단 시점 파악
    ↓
코디네이터: 중단 시점부터 재개
    ↓
개발자: 작업 계속
    ↓
문서가: 진행 상황 업데이트
```

## 🎯 효율적인 활용 팁

### 1. 명확한 지시어 사용
- 역할명을 명시하여 지시 (예: "개발자: 버그 수정")
- 약어 사용으로 토큰 절약 (예: "PC: 작업 시작")

### 2. 적절한 역할 분배
- 복잡한 작업은 여러 역할에 분배
- 단순한 작업은 단일 역할로 처리

### 3. 협업 체계 활용
- 순차적 작업 (분석 → 개발 → 리뷰 → 테스트)
- 병렬 작업 (개발자 + 문서가 동시 작업)

### 4. 필독 문서 참조
- 각 역할은 자신의 필독 문서를 참고
- 최신 정보 유지
